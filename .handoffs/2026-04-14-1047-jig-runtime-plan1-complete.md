## Decisions made

- **No `ToolRegistry` class.** `McpServer` already owns the name → tool map and lazily wires `tools/list` / `tools/call` on the first `registerTool` call (SDK `index.mjs:1335`). The plan's Task 4.3 sketch of a custom registry was redundant and is skipped.
- **Lean result types carry an index signature.** `ToolCallResult` in `src/runtime/handlers/inline.ts` declares `[key: string]: unknown` so it is structurally assignable to the SDK's `CallToolResult`. Same pattern will repeat for resource/prompt result types later.
- **SDK adapter stays quarantined to two files.** `src/runtime/server.ts` and `src/runtime/transports/stdio.ts` are still the only files that import `@modelcontextprotocol/*`. Handlers and `index.ts` are SDK-free.

## Landmines

- **Plan 1's plan doc is outdated from Phase 3 onward.** The Task 4.3 `ToolRegistry` sketch, the `onToolsList` / `onToolsCall` method names, and the handler-arity assumptions in the examples are all pre-SDK-2.x. The runtime's actual shape is `server.registerTool(name, spec, handler)` returning a `RegisteredTool`. Plan 2 should be written against the current code, not against Plan 1's sketches.
- **`npm test` green ≠ all green.** Typecheck catches structural issues JSON equality hides — specifically, SDK types with index signatures (`CallToolResult` is one, more will appear for resources / prompts) reject lean subtypes that lack one. Always run `npm run check && npm test && just smoke` before claiming ready-to-merge.
- **`tools/list` / `tools/call` handlers are lazy.** The SDK only wires them on the first `registerTool` call. Integration tests that skip `registerTool` see `result: undefined` for `tools/list` — not an error, just a silent no-op. Any test for new tool behavior must register at least one tool.
- **Plan 1 handlers ignore args.** Inline handlers return the configured text verbatim regardless of `tools/call.arguments`. `tools/list` still advertises the `input` schema so clients can construct valid calls, but don't expect arg-shaped behavior until Plan 2 lands Mustache / JMESPath.
- **`just smoke` depends on `jq`.** macOS doesn't ship `jq` by default; `brew install jq` is assumed. If Plan 6's CLI work moves the smoke path into a platform-neutral runner, revisit.
- **Integration tests need `{ timeout: 10_000 }`.** Node's test runner has no default per-test timeout; subprocess-based tests hang forever on bugs. See `tests/integration.test.ts` for the established pattern.
- **`.bito.yaml` and `commit.txt` are globally gitignored.** Agent writes `commit.txt`; Clay runs `gtxt` (`git commit -F commit.txt && rm commit.txt`). `.bito.yaml` is regenerated by the building-in-the-open plugin session-start hook — if it vanishes mid-session, restart.
